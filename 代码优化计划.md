# NetWise 代码优化计划

本文档整理自对 **README.md**、**操作文档.md**、**需求分析文档.md** 与当前源码的对照分析，作为后续迭代的 backlog，可按优先级分批实施。

---

## 1. 架构与可维护性

### 1.1 `App.tsx` 体积过大（约 3900+ 行）【部分已优化】

**状态**：**部分已优化**（主文件已显著瘦身；导入主流程、设置与全部 Modal 仍在 `App.tsx`，后续可继续拆 `screens/` / hooks）。

**现状**：解锁、导入、解析、设置、多个 Modal、向导、样式等集中在一个文件。

**问题**：与文档中「可维护性」目标冲突，Code Review、热更新冲突、新人上手成本高。

**建议**：

- 拆分为 `screens/`、`hooks/useImportFlow`、`hooks/useTrendData`、`components/settings/*` 等；
- `App.tsx` 仅负责组合与顶层状态编排。

**优先级**：高（长期收益最大）。

**实施记录（已做）**：

- 新增 `src/app/AppStyles.ts`：`StyleSheet` 全量迁出，`App.tsx` 仅 `import { styles }`。
- 新增 `src/app/homeUiConstants.ts`：资产类 / 趋势 / 平台 / 页面类型 / OCR 作用域 / `MODULE_HINT_TEXT` 等常量。
- 新增 `src/app/importSnapshot.ts`：导入编辑类型、`validate*`、`buildImportSnapshotPayload`、`inferMimeFromUri`。
- 新增 `src/app/formatOcrRuleScopeLabel.ts`：规则作用域展示文案。
- 新增 `src/app/components/AppLockGate.tsx`：安全初始化与口令 / 生物识别解锁 UI。

## 2. 性能

### 2.1 【已优化】同一刷新内多次 `readStore()`（重复解密 + 读盘）

**状态**：**已优化**（代码已合入；行为与优化前一致，仅减少单次刷新内的读盘/解密次数）。

**现状（优化前）**：`refreshTrendData` 使用 `Promise.all` 并行请求主趋势、三平台、各自定义模块等；`assetHistoryDb.ts` 中多数 `query*` 各自调用 `readStore()`，一次 UI 刷新会**多次解密同一加密文件**。

**问题**：历史快照增多时耗电、卡顿风险上升。

**建议（已落实）**：

- 提供单次读盘入口（如内部 `readStoreOnce` 或对外的 `queryAllTrendBundles(store)`）；
- 一次 `readStore` 后在内存中计算主图、分平台、自定义模块所需序列。

**优先级**：高（改动相对集中，收益明显）。

**实施记录**：在 `src/storage/assetHistoryDb.ts` 中抽取 `query*FromSnapshots` 同步计算，新增 **`queryTrendDashboardBundle`**（单次 `readStore`）；`App.tsx` 的 **`refreshTrendData`** 改为只调用该接口。原 `queryStoredMainTrendAndHeroSummary` / `queryPlatformTrendSeriesFull` / `queryCustomRecognitionTrendSeriesFull` 仍保留，内部各 `readStore` 一次后委托给 FromSnapshots，供其它单点调用场景使用。

---

### 2.2 【已优化】趋势计算随历史规模增长

**状态**：**已优化**（主资金趋势 + 主图「全部」分项：由「每个图表日全量扫快照」改为按时间顺序单次扫描，在内存中维护各平台 / 各自定义模块的「截至该日最近一条」指针；复杂度由约 **O(D×N)** 降为 **O(N+D)**，D 为不重复导入日、N 为有资产的快照数。三平台 / 自定义模块子图本身已是按快照单次聚合，未改语义。）

**现状（优化前）**：主趋势对每个图表日构造 `importDate <= date` 的快照池再跑合并，分项再对每个资产类重复该过程。

**建议**：增量扫描、按日期索引；中长期可按 README「下一步」评估 **SQLite / 结构化存储** 与迁移脚本。

**实施记录**：`src/storage/assetHistoryDb.ts` 中新增 **`computeMainTrendCumulativeByClassSeries`**；**`queryStoredMainTrendAndHeroSummaryFromSnapshots`** 一次计算同时生成主折线与 **`mainBreakdown`**；**`findLatestCustomModuleSnapshotInList`** 与增量逻辑共享 **`snapshotQualifiesAsCustomModuleLatestCandidate`**。

**优先级**：中（数据量上来后再做也可）。

## 3. 安全与隐私

### 3.1 【已优化】OCR 全量 `console.log` 无环境门控

**状态**：**已优化**（`App.tsx` 中全量 OCR 文本日志仅在 `typeof __DEV__ !== "undefined" && __DEV__` 时输出；Release 构建不打印正文。）

**现状（优化前）**：`App.tsx` 中存在 `[OCR_FULL_TEXT_BEGIN]` 一类全量 OCR 日志。

**问题**：生产环境若仍输出，系统日志可能泄露金融原文，与 README「发布前关闭 OCR 日志」不一致。

**建议（已落实）**：`if (__DEV__) { ... }`；与 `assetHistoryDb` 中 dev 判断风格一致，使用 `typeof __DEV__ !== "undefined" && __DEV__`。

**优先级**：高（隐私与上架风险）。

---

### 3.2 加密实现可加固

**现状**：`appSecurity.ts` 中 IV 由密钥材料确定性派生，CBC 下对重复结构不够理想。

**建议**：随机 IV + 密文前缀存储；增加版本字段便于历史数据迁移。

**优先级**：中（安全审计或上架前重点）。

---

### 3.3 `readStore` 解密失败时静默返回空数组

**现状**：解密失败时 `console.warn` 后返回 `{ snapshots: [] }`。

**问题**：用户易误解为「数据没了」；后续写入可能覆盖损坏文件。

**建议**：备份损坏文件、明确 Alert、禁止在无用户确认下用空数据覆盖原文件。

**优先级**：高（数据可靠性）。

---

## 4. 可靠性与数据

### 4.1 整文件写入缺少原子性

**现状**：`writeStore` 直接覆盖目标文件。

**问题**：写入过程中崩溃可能导致文件损坏。

**建议**：先写临时文件再替换（同目录 `rename`），或等价原子写入策略。

**优先级**：中。

---

### 4.2 自定义规则 / 模块明文 JSON

**现状**：`netwise-ocr-custom-rules.json` 与 `netwise-custom-recognition-modules.json` 为明文（文档已说明）。

**建议**：若威胁模型包含 root/备份导出，可对配置做轻量加密或与快照一致的密钥策略；并在隐私说明中写清边界。

**优先级**：低～中（视产品定位）。

---

## 5. 工程化与仓库卫生

### 5.1 缺少自动化测试

**现状**：仓库内未见 `*.test.ts` / `*.spec.ts`。

**建议**：对 `parseOcrText`、去重与分桶、`combinedSummary` / 趋势口径等增加单元测试，与需求文档验收要点对齐。

**优先级**：中。

---

### 5.2 `expo-sqlite` 依赖未使用

**现状**：依赖在 `package.json` 中，业务快照未走 SQLite。

**建议**：删除依赖减少体积与误导；或按产品路线落地 SQLite 并迁移。

**优先级**：低（与 README 已对齐说明，可择机处理）。

---

### 5.3 `package.json` 启动脚本写死本机 `adb` 路径

**现状**：如 `D:\Environment\Android\Sdk\platform-tools\adb.exe`。

**问题**：换机器即失效，不利于协作。

**建议**：用 `ANDROID_HOME` / `PATH` 解析 `adb`，或文档中强制说明「需本地修改脚本」。

**优先级**：低～中。

---

## 6. 产品与文档对齐

### 6.1 换机 / 数据仅本地（需求 S5）

**现状**：需求文档写明跨设备迁移不在 MVP。

**建议**：设置或首次引导中增加简短提示：「数据仅保存在本机，卸载或清除数据将丢失」，降低预期偏差。

**优先级**：低。

---

### 6.2 调试入口与测试数据（CFG-4 / CFG-5）

**现状**：已有测试数据写入、调试导出等能力。

**建议**：发布渠道包中确保调试能力不可达或需强确认（如隐藏手势、二次弹窗），避免日常误触污染数据。

**优先级**：中（发布前 checklist）。

---

## 7. 建议实施顺序（摘要）

| 顺序 | 项 | 理由 |
|------|----|------|
| 1 | 单次读盘 + 批量计算趋势（§2.1）**【已优化】** | 收益/改动比高，直接改善卡顿与耗电 |
| 2 | OCR 日志 `__DEV__` 门控（§3.1） | 隐私与合规 |
| 3 | 解密失败不静默丢数据（§3.3） | 避免用户数据「假丢失」与覆盖损坏文件 |
| 4 | 继续拆分 `App.tsx`（§1.1，**部分已完成**） | 长期维护与协作 |
| 5 | 其余条目按版本节奏排期 | 见各节优先级 |
