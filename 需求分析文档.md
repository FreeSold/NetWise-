# NetWise 需求分析文档

## 1. 文档说明

- **项目名称**：NetWise（包名 `com.wanga.netwise`，npm 名 `netwise-mobile`）
- **形态**：移动端（Expo + React Native），Android 为主场景
- **本文目的**：从业务与系统角度归纳 **目标用户、问题域、功能需求、非功能需求与数据边界**，便于产品延续开发与验收对齐。

---

## 2. 背景与问题陈述

### 2.1 背景

个人资产分散在 **招商银行、支付宝、微信** 等 App 的不同页面截图中。用户希望 **少手工录入**，通过 **截图 → 自动识别 → 归类汇总 → 看趋势**，在 **本地** 完成，避免敏感截图与资产明细上传云端。

### 2.2 核心问题

1. 如何从截图中 **可靠提取文本**（OCR）？
2. 如何 **判定页面类型** 并套用 **结构化解析规则**？
3. 如何允许用户 **纠正识别结果** 再落库？
4. 如何在 **保护隐私** 前提下 **持久化** 并支持 **历史趋势**？
5. 内置模板无法覆盖的页面，如何 **扩展**（自定义规则 / 自定义模块）？

---

## 3. 目标用户与使用场景

### 3.1 目标用户

- 有 **定期截图** 理财/钱包/财产页习惯的个人用户；
- 接受 **本地 App** 处理金融截图，不希望数据上云；
- 具备基本智能手机操作能力；开发者用户会参与 **规则调试** 与 **自定义扩展**。

### 3.2 典型场景

| 编号 | 场景 | 期望结果 |
|------|------|----------|
| S1 | 用户截取支付宝理财页，导入 App | 识别为对应页面类型，列出金额项与分类，可编辑后确认入库 |
| S2 | 用户一天内多次导入 **同一张** 截图 | 当日 **去重**，不产生重复快照记录 |
| S3 | 用户连续多日导入 | 首页总资产与趋势图随 **已确认快照** 累积变化 |
| S4 | 内置模板不认识的页面 | 可通过 **自定义 OCR 规则** 或 **自定义识别模块**（关键词命中即把当次导入总额计入模块趋势）扩展 |
| S5 | 用户更换手机或清理数据 | 当前设计以 **单机本地** 为主；跨设备迁移不在 MVP 核心范围内（可另立需求） |

## 4. 功能需求

### 4.1 安全与访问控制

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| SEC-1 | 首次使用强制设置 **6 位数字口令** | 高 |
| SEC-2 | 未解锁前不可查看资产汇总与趋势 | 高 |
| SEC-3 | 支持设备能力范围内的 **生物识别解锁**（与口令体系配合） | 中 |
| SEC-4 | 资产快照持久化内容 **加密**；密钥与口令哈希存 SecureStore | 高 |

### 4.2 导入与 OCR

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| IMP-1 | 支持从 **相册** 选择图片；支持从 **文件** 选择（适配模拟器/弱相册环境） | 高 |
| IMP-2 | 支持 **多图** 批量导入（数量上限以产品定义为准，当前约 6 张） | 高 |
| IMP-3 | OCR 在 **设备本地** 完成（ML Kit），不依赖远程 OCR API | 高 |
| IMP-4 | 对相册/文件权限缺失时给出明确提示 | 高 |

### 4.3 解析与页面识别

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| PAR-1 | 内置支持一批金融 **页面模板**（关键词 + 行解析逻辑），当前包括：招行财产/理财、支付宝理财/基金、微信钱包等 | 高 |
| PAR-2 | 输出统一模型：**金额名称、金额、币种、资产分类、置信度、规则摘要** 等 | 高 |
| PAR-3 | 解析结果可 **人工编辑**：改名称、金额、分类；增删行 | 高 |
| PAR-4 | 可提供 **OCR 原文** 展示供核对（可选展开） | 中 |
| PAR-5 | **自定义 OCR 规则**：锚点原文子串 + 锚点后首数字金额 + 资产类 + 可选页面作用域（含自定义模块作用域） | 中 |
| PAR-6 | **自定义识别模块**：多关键词「或」匹配合并 OCR；命中则将 **当次导入解析总额** 计入该模块趋势；向导式创建 | 中 |

### 4.4 持久化与统计

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| STO-1 | 用户点击确认后写入 **当日快照**；快照含扁平资产列表及 **分桶**（按 screenType 或自定义模块 id） | 高 |
| STO-2 | **同一自然日 + 同一图片 hash** 不重复记录 | 高 |
| STO-3 | 确认后 **释放** 当前导入图片预览与内存引用；**不** 将原始截图文件写入资产历史存储 | 高 |
| STO-4 | 首页 **「目前为止总资产」** 与 **主趋势折线** 数据口径一致（基于同一快照集合计算） | 高 |
| STO-5 | **分平台趋势**（支付宝 / 招行 / 微信）与 **自定义模块趋势**；各卡片可独立选择「全部」或按资产类筛选 | 高 |
| STO-6 | 主图在「全部」下可展示 **按资产类的分项折线**（有数据时） | 中 |

### 4.5 设置与数据维护

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| CFG-1 | 自定义 OCR 规则、自定义模块的 **持久化文件** 可读写；规则列表支持增删改 | 高 |
| CFG-2 | 自定义模块支持 **隐藏/显示** 折线图（不删历史） | 中 |
| CFG-3 | 提供 **清除今日导入**、**清空全部导入**（明确不删配置） | 中 |
| CFG-4 | **测试数据** 写入与清除路径（仅调试），与真实数据区分标记 | 低（开发/测试） |
| CFG-5 | 调试能力：如导出/展示 **已存快照 JSON**、折线数据结构等（受开关控制） | 低（开发） |

### 4.6 内置页面范围（当前批次）

以下页面为 README 与解析层已对齐的 **第一批** 支持对象（随版本迭代可增删）：

- 招商银行：财产页、理财页  
- 支付宝：理财页、基金页  
- 微信：钱包页  

未命中内置模板时，可走 **unknown + 自定义规则/自定义模块** 路径。

## 5. 非功能需求

| 类别 | 要求 |
|------|------|
| 隐私 | 截图 OCR 本地处理；快照加密；原始截图不落库 |
| 性能 | 单批多图 OCR 在可接受时间内完成（与机型相关，不设具体 SLA）；**已优化**：首页趋势刷新合并为单次加密快照读盘与内存批量计算（`queryTrendDashboardBundle`，见《代码优化计划》§2.1【已优化】） |
| 可维护性 | TypeScript 严格类型；解析规则集中在 `templates.ts` / `shared.ts` |
| 可扩展性 | 页面模板可增；自定义规则与模块文件化，便于备份（用户需自行管理文件级备份若未来支持） |
| 合规提示 | 金融数据敏感；用户需理解数据仅存本机，卸载或清数据会丢失 |

## 6. 数据与领域模型（摘要）

### 6.1 核心实体

- **ParsedAsset**：单条资产（名称、金额、分类、来源页面类型等）
- **ParseResult**：单次 OCR 的解析结果（页面类型、资产列表、警告、可选展示标签、命中的自定义模块 id 列表等）
- **Snapshot**：一次用户确认后的持久化单元，包含日期、图片 hash 列表、资产列表、分桶、（新数据）各图 OCR 全文用于模块匹配等

### 6.2 资产分类枚举

现金、基金、保险、股票、理财（`types.ts` 中与界面一致）。

### 6.3 存储介质说明

- **资产快照**：加密 JSON 文件 `netwise-asset-history.json`（实现于 `assetHistoryDb.ts`）。  
- **未使用 SQLite**：`package.json` 中不包含 `expo-sqlite`；若未来迁移为库表存储，需加回依赖并单独修订本需求与操作文档。

## 7. 约束与已知限制

1. **模板强依赖 OCR 质量**：模糊截图、异形字体、深色模式可能导致识别失败或页面类型误判。  
2. **自定义规则为子串 + 后置首数字**，复杂版式可能误匹配，需靠「限定页面」降误报。  
3. **单机本地**：无账号体系则无官方云同步。  
4. **生物识别** 依赖系统策略与硬件。  
5. **Dev Client / 原生模块**：OCR 与 SecureStore 等需正确原生构建链。

## 8. 后续需求方向（建议 backlog）

1. 更丰富的内置银行/券商页面模板与回归测试集。  
2. 可选的数据 **导出/备份/恢复**（加密包）。  
3. 解析规则 **可视化测试台**（贴 OCR 文本即出 ParseResult）。  
4. iOS 真机一致性与上架合规清单。  
5. 若坚持 SQLite：将快照迁移为表结构 + 迁移脚本，与现有加密 JSON 方案做兼容或一次性导入。

---

## 9. 验收要点（摘要）

- 首次启动 → 设口令 → 重启 → 解锁 → 可见主界面。  
- 导入内置模板截图 → 解析合理 → 编辑一行 → 确认 → 首页数字与趋势更新。  
- 同日同图二次确认 → **不** 重复增加记录。  
- 自定义规则在限定 scope 下命中/不命中符合预期。  
- 自定义模块关键词命中后，模块折线金额与当次导入总额逻辑一致。  
- 清空全部导入后趋势为空，但自定义配置仍在。
